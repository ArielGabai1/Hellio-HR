Skill matching rules:
- candidate_skills has columns (candidate_id, skill) -- skill is VARCHAR text, NOT an ID.
- position_skills has columns (position_id, skill) -- skill is VARCHAR text, NOT an ID.
- There is NO "skills" lookup table. NEVER JOIN to a "skills" table.
- For exact skill matching: LOWER(cs.skill) = 'skillname' (NOT ILIKE).
- Use ILIKE '%skill%' ONLY when the user asks for broad/partial skill matches.

Table selection rules:
- position_requirements stores free-text job requirements with a type column ('required', 'nice_to_have').
- For "tech stack" or "technologies" questions about positions: JOIN position_skills directly.
- For "requirements" or "qualifications" questions: query position_requirements.
- Do NOT JOIN with candidate_skills/position_skills unless the question specifically asks about skills or tech stack.
- Do NOT JOIN with certifications/education/experience unless the question asks about those.
- For experience_level, status, location, or name queries, query candidates/positions directly.

Location matching rules:
- Location names may contain apostrophes (e.g., Be'er Sheva, Ra'anana). Use double-apostrophe escaping: ''
- For location queries, ILIKE is acceptable since locations may have suffixes like ", Israel".
- Note: ILIKE '%tel aviv%' will also match "Remote or Tel Aviv office" -- this is acceptable behavior.

Negation queries:
- For "candidates who do NOT have skill X": use NOT EXISTS or NOT IN subqueries.
- Do NOT use LEFT JOIN with WHERE != for negation -- it produces wrong results.

Aggregation rules:
